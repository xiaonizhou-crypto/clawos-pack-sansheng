# HIGH-STAR README & MARKETING LESSONS

> 研究对象（抽样）：React、Supabase、LangChain、n8n、Appwrite、Streamlit。  
> 目标：提炼高星 GitHub 项目在 README / 首发传播上的共性，并转成三省六部项目可直接借鉴的动作。

---

## 一、我看到的共性

### 1. 第一屏一定极快回答“这是什么”
高星项目很少在第一页拐弯。
它们通常都会非常快地说清：
- 产品是什么
- 给谁用
- 为什么值

比如：
- React：一行定义 + 3 条核心价值
- n8n：一句定位 + 一张大图 + key capabilities
- LangChain：一句平台定义 + quickstart

### 对三省六部的启发
不能只靠世界观钩子。
必须始终同时说清：

> 三省六部是一个 **可运行的 AI 组织系统 / roster system**。

而“朝廷”是它的记忆点，不是唯一解释路径。

---

### 2. 高星项目很早就放真实界面图
Supabase、n8n、Appwrite 都很典型：
- 标题之后不久就放 dashboard / product screenshot
- 不是等讲完原理才给图

### 对三省六部的启发
这次我们把：
- roster-overview
- dashboard-overview
- employee-marketplace
放进 README，是对的。

应该继续坚持：

> **真实界面图越早出现，可信度越高。**

---

### 3. 高星项目很快给“怎么开始”
LangChain、React、n8n 都一样：
- quickstart 很靠前
- 安装命令很明确
- 用户不需要翻很久才知道下一步

### 对三省六部的启发
我们的 `30 秒安装` 保持靠前是对的。
后面不要把它继续往后挪。

---

### 4. 高星项目会把“核心价值”抽成短列表
常见结构：
- 3~6 个 bullets
- 每条一句
- 强调结果，不强调空泛理念

### 对三省六部的启发
我们后续可以更明确抽成：
- 有入口中枢
- 有审核之门
- 有真实执行
- 有 roster / hiring / 扩编能力

而不是只讲沉浸感。

---

### 5. 高星项目会区分“主页面”和“深度资料”
它们不会把所有内容都塞首页。
通常会把：
- 介绍
- quickstart
- 关键图
放主 README
而把：
- 架构
- 深度说明
- 角色细节
放文档页或次级资源

### 对三省六部的启发
这是我们这轮把角色大表后移的原因，也是正确方向。
如果还继续收口，甚至可以考虑：

> 把角色阵容的更重内容进一步交给 `ROLE_ROSTER.md`

让主 README 更像商品页。

---

### 6. 高星项目很擅长“产品 + 生态 / 扩展性”一起讲
比如：
- LangChain 会讲 ecosystem
- Streamlit 会讲 gallery
- n8n 会讲 templates / integrations

### 对三省六部的启发
我们的“员工市场 / 扩编 / 角色补位”其实就是生态入口。
这块很值钱，不该只是边角功能。

对外可讲成：
- roster system
- expandable organization
- hireable roles
- modular presets

---

### 7. 高星项目很少把自己写成纯概念作品
哪怕它们愿景很大，README 也会尽量让人觉得：
- 这是个能装的东西
- 这是个能跑的东西
- 这是个现在就能试的东西

### 对三省六部的启发
我们后续所有传播，都要尽量避免写成：
- 世界观小说
- 角色设定展
- 美术设定集

而应该保持：

> **这是一个真的可以安装、验证、派活、扩编的组织系统。**

---

## 二、可直接借给三省六部的打法

### 打法 1：双层定位
推荐长期保留这两个层次：

#### 层 1：一眼看懂版
三省六部是一个可运行的 AI 组织系统。

#### 层 2：记忆点版
它不是安装一个 bot，而是召来一整座替你办事的朝廷。

这样同时兼顾：
- 易懂
- 记忆点

---

### 打法 2：截图不是配菜，是信任锚点
推荐长期保留：
- roster-overview
- dashboard-overview
- employee-marketplace

并把它们视为：
- README 主资产
- Release 主资产
- 社媒传播素材基础

---

### 打法 3：Quickstart 必须稳定靠前
对三省六部这种带世界观的项目尤其重要。
因为如果安装太靠后，用户会误以为它只是概念包。

---

### 打法 4：把“组织能力”讲得比“古风感”更硬
未来传播时，优先讲：
- roster
- assign task
- review gate
- execution chain
- expand / hire
- preset install

古风感负责让它被记住，
组织能力负责让它被相信。

---

### 打法 5：主 README 卖产品，次级文档卖世界深度
推荐长期分工：
- `README.md`：卖产品
- `ROLE_ROSTER.md`：卖角色
- `PRODUCT_STORY_SPINE.md`：卖叙事
- `FIRST_RELEASE_NOTES.md` / `RELEASE_PAGE_DRAFT.md`：卖首发

---

## 三、对三省六部的直接建议

### 最应该继续加强的 3 个点

#### 1. 把一句现代定位再写得更硬一点
可以考虑在 README 前部补一句：

> 三省六部是一个支持 roster、任务派发、审核封驳与扩编补位的 AI 组织系统。

#### 2. 把截图 caption 继续往“产品能力”方向压
少一点纯氛围，多一点可执行价值。

#### 3. 后续如果发 GitHub / 社媒，优先发“产品能力图”而不是纯世界观海报
因为高星项目最会做的，不是只让人觉得酷，
而是：

> **让人 10 秒内理解它有多能打。**

---

## 四、结论

高星 GitHub 项目真正厉害的，不是文案更花，
而是它们把这三件事同时做到了：
- 一眼懂
- 一眼真
- 一眼知道怎么开始

对三省六部来说，最强传播路线也应该是：

> **组织系统先让人理解，朝廷世界观再让人记住。**
